## Problem Statement

A bug is found in production days after a feature shipped. The developer asks: "What agent session wrote this code, what was it thinking, what spec was it working from, and was the spec even clear?"

Today this question is unanswerable. The data exists in three disconnected systems:
- **git-ai Git Notes**: which agent session wrote which lines of which commit
- **agentlytics cache**: full session transcripts, tool calls, file accesses, timestamps
- **Work-item systems** (Linear, Jira, GitHub): which issues, specs, and plans the work was for

Nobody joins these data sources. git-ai knows authorship but not intent. agentlytics knows what the agent did but not why. Linear knows the work item but not the agent. Carabiner is the join.

## What Makes This Cool

`carabiner why src/auth/handler.go:47` returns a causal dossier:

```
LINE: src/auth/handler.go:47 (introduced in commit abc1234)
SESSION: Claude Code session 8f3a2b1c (2026-03-28 14:32-15:47)
MODEL: claude-sonnet-4-5-20250514
CONFIDENCE: high (deterministic join via git-ai)

WORK ITEM: ENG-42 "Implement token refresh flow"
  Source: branch name (feature/ENG-42)
  Spec: docs/designs/auth-refresh.md (linked in issue)

SESSION CONTEXT:
  The agent was implementing token refresh per the spec. It chose to
  cache the refresh token in-memory rather than using the session store.
  The spec was ambiguous on storage location (line 34: "persist the token").

SUBSEQUENT TOUCHES:
  - Commit def5678 (2026-03-29): human edit, no agent session
  - Commit ghi9012 (2026-03-30): Codex session 4e7f9a2b, same work item
```

The "whoa" moment: `git blame` for intent, not authorship.

## Landscape Context

### Existing Tools (Layer 1 — solved problems)

**agentlytics** (github.com/f/agentlytics): Reads local session stores from every major AI coding agent (~/.claude/projects/, ~/.codex/sessions/, ~/.openclaw/agents/, etc.). Normalizes into ~/.agentlytics/cache.db (SQLite). 30-second scan across all agents. Does collection. Does NOT do attribution or work-item linkage.

**tokscale** (github.com/junhoyeo/tokscale): Rust CLI tracking token usage and cost across agents. Same local data sources as agentlytics. Focused on cost analytics and leaderboards. No attribution.

**git-ai** (github.com/git-ai-project/git-ai): 1,481 stars, daily commits, Apache-2.0. Uses Git Notes (refs/notes/ai) for line-level authorship logs. v3.0.0 spec: session hashes from SHA-256("{tool}:{conversation_id}"), agent_id (tool, id, model), conversation transcripts, line ranges per file. Handles history rewrites (rebase, squash, cherry-pick, amend). Does attribution. Does NOT do work-item linkage.

### Adjacent Tools (Layer 2 — partial overlap)

**Apache DevLake** (devlake.apache.org): Builds PR/issue/repo graphs across GitHub and Jira. Solves a chunk of the work-item normalization problem. Dashboard-oriented, not CLI-first. No AI session awareness.

**GitHub Copilot tracing** (March 2026): Built-in session-to-commit tracing, but Copilot-only. Vendor lock-in.

**Entire CLI** (ex-GitHub CEO): Git observability layer for AI agents. Context management during sessions, not retroactive attribution.

### The Gap (Layer 3 — first principles)

**EUREKA: Everyone builds for explicit attribution (agents must announce themselves) or single-vendor tracing. Nobody does vendor-agnostic retroactive attribution by joining data that already exists across multiple systems. The deterministic join key (git-ai's agent_id.id = agentlytics' conversation_id) makes this a foreign key join, not fuzzy correlation. Session-level retroactive attribution is sufficient for forensic debugging and requires zero agent cooperation beyond what git-ai already handles.**

## Terminology

- **Agent session (agentlytics):** A single conversation between a human and an AI coding agent, stored locally by the agent (e.g., a `.jsonl` file in `~/.claude/projects/`). Identified by a `conversation_id` (typically a UUID).
- **Attribution record (git-ai):** A Git Note attached to a commit at `refs/notes/ai`, containing an authorship log that maps file paths and line ranges to session hashes. The session hash is `SHA-256("{tool}:{conversation_id}")[:16]`. The metadata section includes `agent_id.id` which is the raw `conversation_id` before hashing.
- **Coding session (human concept):** The period of time a developer works with an agent on a task. May span multiple agent sessions (agent crashes, restarts) and multiple commits.
- **Join key:** `git-ai.prompts[hash].agent_id.id` = `agentlytics.sessions.conversation_id`. This is the deterministic foreign key that connects attribution to session data. **This join key is unvalidated with real data and must be verified before implementation (see Pre-Implementation Gate below).**

## Premises

1. **Collection is commodity.** agentlytics (and similar) read vendor session stores. Solved problem.
2. **Line-level attribution is commodity.** git-ai handles it via Git Notes. Active project, open standard, expanding agent support. Solved problem.
3. **The join is the product.** Nobody connects git-ai attribution + agentlytics session data + work-item systems into a single queryable chain. That cross-system join is carabiner.
4. **git-ai and agentlytics are foundations, not competitors.** Carabiner reads their data. It doesn't reinvent collection or attribution. Optional dependencies, graceful degradation without either.
5. **Work-item linkage rides existing integrations.** Linear/Jira already link PRs to issues via branch naming and PR integrations. Carabiner reads this graph. The normalization layer (ENG-42 vs PROJ-123 vs #456) is carabiner's added value.
6. **Two installation scopes, one query interface.** System-level reads agentlytics. Repo-level reads git-ai + git history + work-item refs. Both feed the same SQLite. `carabiner why` works regardless of which data sources are available.
7. **Graceful degradation determines confidence.** git-ai installed -> high confidence (deterministic hash join). No git-ai -> medium (timestamp + file Jaccard). No agentlytics -> partial (git-ai notes only, no session transcripts). Nothing -> carabiner has nothing to join.
8. **Confidence is core product, not UI garnish.** (from Codex) A bug is often many-to-many: one session wrote code, another edited it, a human reviewed it, the spec drifted. If carabiner presents a clean narrative when evidence is messy, trust erodes. Every hop in the chain carries a confidence label.

## Cross-Model Perspective

Codex reviewed this architecture independently. Key contributions:

- **"git blame for intent, not authorship"** — better framing than "forensic tracing." The command should be `carabiner why`, not `carabiner trace`.
- **Deterministic join is the spine** — everything else is enrichment on top of agent_id.id = conversation_id.
- **Apache DevLake** — already builds the PR/issue/repo graph across GitHub and Jira. Study it for work-item normalization rather than building from scratch.
- **Critical risk: many-to-many attribution.** The confidence model is the product. Carabiner must be honest about ambiguity, not tell clean stories from messy evidence.
- **Weekend MVP: one killer query end-to-end.** Prove the join works before building infrastructure.

## Approaches Considered

### Approach A: "One Killer Query" — MVP
Ship exactly one command: `carabiner why <file>:<line>`. Reads git-ai notes, agentlytics cache, and git history on demand. No indexing step, no daemon, no config. Auto-detects available data sources on first run. Returns a causal dossier with confidence per hop. Parses work-item refs from branch names and commit messages (no API calls).

- **Effort:** S-M (CC: ~1 hour, contingent on Proof of Join validation)
- **Risk:** Low (if join key validates), High (if it doesn't)
- **Pros:** Single-command magic moment. Zero config. Proves the join works.
- **Cons:** No aggregate views, no audit by work item, no reporting.

### Approach B: "Index + Query" — Fast Follow
`carabiner index` builds a local SQLite joining git-ai notes + agentlytics sessions + work-item refs. `carabiner why` queries the index. `carabiner audit <work-item>` shows all sessions and commits for an issue. `carabiner report` gives the aggregate view. Index triggered by git hooks (post-commit) or manual.

- **Effort:** M (CC: ~1 hour)
- **Risk:** Medium
- **Pros:** Supports all three value layers (forensics + audit + analytics).
- **Cons:** More surface area. Index freshness becomes a concern.

### Approach C: "Adapter Pattern" — Deferred
Define a DataSource interface. Ship adapters for git-ai, agentlytics, git-history. Future adapters for Linear API, Jira API, DevLake plug in without changing core logic.

- **Effort:** M-L (CC: ~1.5 hours)
- **Risk:** Medium-high
- **Pros:** Most extensible. Community can add adapters.
- **Cons:** Premature abstraction. Three adapters don't need an interface yet.

## Recommended Approach: A then B

Ship `carabiner why` first. Prove the join works end-to-end with real data. If the query feels magical, the product is real. Add indexing and aggregate queries (Approach B) when on-demand queries get slow or when the audit/reporting use cases are needed.

Approach C is deferred until there are 5+ data sources justifying the abstraction.

### MVP CLI Surface

```
carabiner why <file>:<line> [--rev <commit>]
  Returns: causal dossier (session, model, work item, spec, confidence per hop)

carabiner doctor [--json]
  Returns: detected data sources and expected confidence level
  Exit codes: 0 = all sources found, 1 = partial (some missing), 2 = no sources
```

Example `carabiner doctor` output:

```
carabiner doctor v0.1.0

  [ok] git repo          .git/ detected
  [ok] git-ai notes      refs/notes/ai found (47 commits annotated)
  [ok] agentlytics       ~/.agentlytics/cache.db (schema v2, 183 sessions)
  [--] GitHub CLI         gh not found (work-item resolution from PRs disabled)

EXPECTED CONFIDENCE: high
  Line -> commit:    deterministic (git blame)
  Commit -> session: deterministic (git-ai join key)
  Session -> data:   full (agentlytics available)
  Commit -> work item: branch name parsing only (no gh CLI)

RECOMMENDATION: install GitHub CLI for richer work-item resolution
```

With `--json`, returns structured output for programmatic use by agents.

### Reference: agentlytics Actual Schema (cache.db) — VALIDATED 2026-04-04

Verified against real agentlytics cache.db (155MB, multi-agent data).

**Tables:** `chats`, `messages`, `chat_stats`, `tool_calls`, `meta`

**Required contract for carabiner (chats table):**
- Table: `chats`
- Column: `id` (TEXT PRIMARY KEY) — the raw UUID matching git-ai's `agent_id.id`. **VALIDATED: exact match confirmed on commit 733241d.**
- Column: `source` (TEXT) — agent identifier ("claude-code", "codex", "opencode", "cursor", "gemini-cli", "vscode")
- Column: `folder` (TEXT) — working directory / repo path
- Column: `created_at` / `last_updated_at` (INTEGER) — epoch milliseconds
- Column: `name` (TEXT) — session name/description
- Accessible via: `~/.agentlytics/cache.db`

**Enrichment tables:**
- `messages`: full conversation transcript (role, content, model, token counts) keyed by `chat_id`
- `chat_stats`: aggregate stats (token totals, message counts, models used) keyed by `chat_id`
- `tool_calls`: individual tool invocations (tool_name, args, timestamp) keyed by `chat_id`

**Note on ID formats by agent:**
- Claude Code: standard UUID (`655eb4a6-822f-4d52-8c06-cade4afdcd8d`)
- OpenCode: prefixed ID (`ses_2a8b35240ffelYkXF3bnAXy0ua`)
- Other agents: varies, but git-ai's `agent_id.id` should match

If agentlytics schema violates this contract at runtime, `carabiner doctor` reports the mismatch and `carabiner why` degrades to git-ai-only output.

### The Join Algorithm (MVP)

```
Input: file path + line number + optional revision

Step 1: git blame -> commit SHA that introduced the line
  - Use git blame with -C (detect copies) for rename awareness
  - If blame returns a merge commit: follow the parent that introduced
    the change (use git blame -C on the merge commit's diff)
Step 2: read git-ai Git Note for that commit (refs/notes/ai)
  - Parse attestation section: file path -> session_hash -> line ranges
  - Confirm our target line falls within an AI-authored range
  - Parse metadata section: extract agent_id.id, agent_id.tool, agent_id.model
  - If commit was rebased/squashed: git-ai handles rewriting notes to new SHAs,
    so the note should exist on the current commit. If missing, the note was
    lost during a history rewrite git-ai didn't handle.
  - If no git-ai note: fall back to timestamp + file overlap (medium confidence)
Step 3: query agentlytics cache.db for conversation_id = agent_id.id
  - Query: SELECT * FROM sessions WHERE conversation_id = ?
  - Extract: session timeline, files accessed, tool calls, token usage
  - If schema mismatch or no agentlytics data: report git-ai data only (partial)
Step 4: extract work-item references
  - Parse branch name for issue key patterns (ENG-42, PROJ-123, #456)
  - Parse commit message for issue references
  - Parse PR title/description if available (gh pr list for the commit)
  - Normalize to carabiner work-item ref format
Step 5: find spec references
  - Check work-item description for doc links
  - Check commit trailers for spec refs
  - Check session transcript for file reads of .md design docs
Step 6: find subsequent touches
  - git log --follow for the file, filter commits after the introducing commit
  - For each, repeat steps 2-4 to build the touch chain
Step 7: assemble dossier with confidence per hop
  - Each hop: data_source, confidence (high/medium/low/missing), evidence
```

### Edge Cases

| Case | Handling |
|------|----------|
| Merge commit in blame | Follow the parent that introduced the change via `git blame -C` |
| Rebased/squashed commit | git-ai rewrites notes to new SHAs; if note is missing, report as "attribution lost during history rewrite" |
| File renamed | `git log --follow` in Step 6 tracks renames; Step 1 uses `git blame -C` for copy detection |
| Line authored by human (no git-ai range) | Dossier reports "human-authored line, no agent session" with commit metadata only |
| Multiple sessions on same line | Show all sessions in chronological order with "introduced by" vs "modified by" labels |

### Fallback Correlation (when git-ai is absent)

When no git-ai note exists for a commit, carabiner falls back to fuzzy matching:

- **Timestamp comparison:** commit author time (not committer time) vs agentlytics session `ended_at`, tolerance window ±15 minutes
- **File overlap:** Jaccard similarity of `files_changed_in_commit` vs `files_accessed_in_session`, threshold > 0.3 for medium confidence
- **Repo path match:** agentlytics session `repo_path` must match the current repo root
- Both conditions (timestamp + Jaccard > 0.3) required for medium confidence. Timestamp alone = low confidence.

### Confidence Model

Every hop in the attribution chain carries a confidence label:

| Hop | High | Medium | Low | Missing |
|-----|------|--------|-----|---------|
| Line -> commit | git blame (deterministic) | — | — | file not tracked |
| Commit -> session | git-ai note (hash join) | timestamp + file Jaccard > 0.3 | timestamp only (±15min) | no data |
| Session -> transcript | agentlytics (conversation_id match) | — | — | no agentlytics |
| Commit -> work item | explicit trailer or PR link | branch name pattern | commit message grep | no reference found |
| Work item -> spec | explicit link in issue | .md file read in session | — | no spec found |

The dossier shows the weakest link. If the overall chain is medium confidence, the output says so and explains which hop is weak.

### Data Sources and Paths

| Source | Path | What it provides |
|--------|------|------------------|
| git-ai Git Notes | `refs/notes/ai` (in repo) | Session hash, agent_id, line ranges, transcripts |
| agentlytics cache | `~/.agentlytics/cache.db` | Full session data: tool calls, file accesses, timestamps |
| Git history | `.git/` (in repo) | Commits, branches, blame, log |
| Branch names | git branch metadata | Work-item refs (ENG-42 pattern) |
| Commit messages | git log | Work-item refs, spec refs, trailers |
| PR metadata | `gh pr list` (GitHub CLI) | PR -> issue links, descriptions |

### What Happens to Existing Carabiner Code

The prior design (donovanyohan-master-design-20260404-080000.md) proposed removing the enforce subsystem and building hooks + session_mappings table. This design supersedes that:

**Still remove:**
- `carabiner enforce` and all enforce/validate code — the enforce subsystem (linting, type-checking, tool config generation) was the original product direction but is orthogonal to the observability mission. With git-ai handling attribution and agentlytics handling collection, carabiner's identity is the join layer. Keeping enforce dilutes that identity and adds maintenance surface for a feature that duplicates what golangci-lint, eslint, etc. already do natively. The enforce templates remain as standalone YAML in docs/ for anyone who found them useful.
- Quality recording subsystem — pattern recording from gate failures is a separate product concern (proactive quality vs retroactive forensics). It may return as a future layer once the join layer is proven, but it competes for implementation attention in the MVP.

**Repurpose:**
- Existing SQLite at `.carabiner/carabiner.db` becomes the index store (Approach B)
- Existing event log schema informs the index schema
- `carabiner init` repurposed: instead of scaffolding enforce configs, it runs `carabiner doctor` and reports available data sources

**New:**
- `carabiner why` command (the product)
- `carabiner doctor` command (diagnostics)
- git-ai reader (parse refs/notes/ai)
- agentlytics reader (query ~/.agentlytics/cache.db)
- Work-item normalizer (branch names, commit messages, PR metadata)
- Confidence model (per-hop labels)

**Deferred (Approach B):**
- `carabiner index` command
- `carabiner audit <work-item>` command
- `carabiner report` command
- Git hook triggers for automatic indexing

### Relationship to Prior Design Decisions

**Hooks (from prior design):** The prompts-as-primary, blocking-as-backstop hook model is no longer the primary attribution mechanism. git-ai handles that. Hooks become an optional enhancement: if carabiner hooks are installed and context is set explicitly, the confidence goes from "medium" (branch name parsing) to "high" (explicit trailer). But the system works without hooks.

**Session mappings table (from prior design):** Superseded by the git-ai join. The session_mappings table was carabiner's attempt to correlate sessions to commits. git-ai does this deterministically. Carabiner reads git-ai's data instead of maintaining its own mapping.

**agentlytics import (from prior design):** `carabiner telemetry import agentlytics` is replaced by direct reads of agentlytics' cache.db. No import step, no copy of data. Carabiner queries agentlytics in place.

## Open Questions

1. **agentlytics schema stability.** We're reading ~/.agentlytics/cache.db directly. If agentlytics changes their schema, carabiner breaks. Should we pin to a known schema version, or vendor a copy of agentlytics' reader code?

2. **git-ai adoption.** The architecture is richest with git-ai installed. If git-ai adoption is low in a given repo, most queries fall back to medium confidence. Is the fallback experience good enough to be useful, or does it feel broken?

3. **PR metadata access.** Parsing branch names for issue keys is fragile (not everyone uses ENG-42 conventions). `gh pr list` requires GitHub CLI and authentication. How much work-item resolution is worth the complexity in the MVP?

4. **Session transcript size.** agentlytics stores full session transcripts. A single Claude Code session can be megabytes. How much of the transcript does `carabiner why` include in the dossier? Summary? Full? Relevant section only?

5. **Multi-agent attribution.** Codex flagged many-to-many: multiple sessions touching the same code. The "subsequent touches" section of the dossier handles this, but the UX of "here are 4 sessions that touched this line over 3 days" needs design work.

6. **Apache DevLake integration.** Codex suggested DevLake for the PR/issue graph. Worth investigating for Approach B, but probably overkill for MVP.

## Success Criteria

1. `carabiner why src/foo.go:42` returns a causal dossier with session, model, work item, and confidence labels in a repo with git-ai + agentlytics installed
2. `carabiner why` degrades gracefully when git-ai or agentlytics is missing, showing what data is available and what's missing
3. `carabiner doctor` accurately detects available data sources and reports expected confidence level
4. The deterministic join (git-ai agent_id.id = agentlytics conversation_id) works for Claude Code and Codex sessions
5. Work-item refs are correctly parsed from branch names following common patterns (ENG-42, PROJ-123, #456)
6. Confidence labels are honest: if the chain is weak, the dossier says so
