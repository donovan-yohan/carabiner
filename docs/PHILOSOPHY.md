# Philosophy

Foundational thinking behind carabiner's design. Reference this when brainstorming features, evaluating competing approaches, or explaining carabiner's position in the agentic coding space.

Source: Slate binary analysis session + adversarial review research + belayer v5 direction (2026-03-29/30)

---

## The Three Roles

Agent, Harness, and Orchestrator are separate concerns. Carabiner is the Harness.

**Agent** = raw model capability. Claude, Codex, Slate, Gemini, whatever comes next. The agent reads code, writes code, runs commands, solves problems. Agent capability is improving rapidly. Neither carabiner nor belayer own this.

**Harness** = the repo's institutional memory for agents. Quality patterns learned from failures. Domain knowledge agents can't derive from code. This is carabiner. Any agent that can run a shell command can use it.

**Orchestrator** = the layer responsible for getting work done end-to-end. Intake (idea to spec), Implementation (agent does the work), Output (review, gates, PR). This is belayer. YAML pipelines, node execution, gate contracts.

The separation test: you can use carabiner without belayer (manually query quality/knowledge before coding). You can use belayer without carabiner (pipelines work, no quality memory). They compose but don't depend on each other.

## Two Jobs: Quality and Knowledge

### Quality (reactive)

Observed feedback from review failures. When an adversarial gate catches a bug, carabiner records the pattern. This is learned, not authored. The data is generated by the system.

Examples:
- "Auth route changes must update the middleware registry. 3 PRs failed for this."
- "New React components without Storybook stories fail review 80% of the time."
- "JSON file writes must be atomic (write-to-temp-then-rename). open('w') truncates before writing."

Quality patterns have file paths, tags, and recommendations. Retrieval is path-based: what files are you touching? Here's what keeps failing when people touch those files.

### Knowledge (proactive)

Domain semantics agents need but can't derive from reading code. Authored by humans or extracted from domain experts. The data is curated, not generated.

Examples:
- "total_spend must subtract CREDIT rows and add DEBIT rows. All amounts stored as absolute values."
- "The 'archived' status in the pipeline means the node completed successfully, not that it was cancelled."
- "Customer IDs in the legacy system are 1-indexed. The new system is 0-indexed. Off-by-one on migration."

Knowledge doesn't decay the same way quality patterns do. "CREDIT rows are subtracted" is true until the schema changes.

## H-as-Feature: Why Context Loss Is Sometimes Desirable

In the D > 4H framing:
- D = information loss from a single long context (dumb zone degradation)
- H = information loss per handoff between pipeline nodes

For adversarial review, H is a feature, not a cost.

LLM sycophancy research shows models perform better at review when they DON'T share context with the implementer. Isolated reviewers catch more issues because they lack implementation bias. They evaluate the artifact on its own merits rather than validating the author's reasoning.

This is why carabiner's quality patterns are intentionally thin. "Auth routes need middleware registry updates" is a fact about what fails review, not an explanation of why the original author made their decision. The reviewer gets the pattern without the implementation context.

---

## Trust Agent Competency

Carabiner doesn't prescribe how agents work. It doesn't tell the agent to "first plan, then implement, then review." The agent decides its own workflow.

Carabiner provides two things: "here's what keeps failing in this repo" (quality) and "here's what you need to know that the code won't tell you" (knowledge). What the agent does with that information is the agent's business.

The previous harness plugin had 12+ commands prescribing a rigid workflow (brainstorm, plan, orchestrate, validate, review, reflect, complete, prune). This fought the natural evolution of models getting better at self-planning and self-reviewing. Carabiner is a knowledge layer, not a workflow engine.

---

## The False Positive Contamination Problem

Gate failures are not ground truth. When a gate says "this auth middleware is missing a registry update," that might be a real bug, a false positive, or a stylistic preference.

If carabiner auto-evolves quality standards from gate failures, and 30% are false positives, it encodes false positives as institutional knowledge. After 6 months, the quality patterns file contains "always do X" where X is something a model hallucinated.

The real ground truth is bugs traced back to specific runs. Code that made it through the gate, got merged, and then caused a problem. Carabiner can tie bugs to specific design docs, branches, and commits because it tracks these artifacts.

Evaluator self-evolution is descoped until this external signal (traced bugs) is reliable. Quality patterns from gate failures are useful but treated as provisional, not authoritative.

---

## Why Not a Plugin

The previous harness was a Claude Code plugin. This locked it to one agent ecosystem. An agent plugin only works in the agent it's installed in.

Carabiner is a CLI. `carabiner quality check --files src/auth/` works from Claude Code, from Codex, from a shell script, from a CI pipeline, from any agent that can run `sh -c`. The universal interface is the shell command.

Carabiner ships lightweight plugins/skills for major agents (Claude, Codex, Gemini, OpenCode) as convenience wrappers. These just teach the agent that carabiner exists and how to call it. The real work happens in the CLI binary.

---

## Concurrent Branch Safety

Quality patterns are repo-level knowledge but gate results happen on branches. With multiple pipelines running simultaneously, mutable counters create race conditions.

Solution: append-only signal files. The learning file is immutable after creation. Each gate result writes its own signal file. Status (active, dormant, stale) is computed at read time from signal history, never stored. No write conflicts, no stale counters.

---

## Relationship to Belayer

Belayer is the orchestrator. It runs YAML pipelines, executes nodes, routes based on gate outcomes. When a gate fails, belayer routes to `on_fail`. Belayer doesn't know WHY it failed and doesn't know about quality patterns.

Frameworks are the glue. The `claude-codex-carabiner` framework (shipped with belayer) includes scripts that call both `belayer` and `carabiner` CLI commands. The framework's `implement.sh` calls `carabiner quality check` before prompting the agent. The framework's gate failure handler calls `carabiner quality record`. This is the framework's decision, not belayer's or carabiner's.

A different framework could skip carabiner entirely. A user could use carabiner without any framework. The separation is real.
